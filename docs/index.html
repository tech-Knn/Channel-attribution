🎯 The Problem

We run AFS (AdSense for Search) across a content network. Google gives us custom channels — identifiers we attach to ad requests so revenue can be tracked. We have approximately 2,000 of these channels.

We publish roughly 10,000 articles per month. Each article should have a channel assigned so we know exactly how much revenue it generates. But:

🔴 Problem 1: Idle Channels

At any given time, hundreds of channels sit idle — disapproved by Google, no longer delivering ads, or simply never assigned to an article. Every hour an idle channel exists, potential revenue is lost.

🔴 Problem 2: Dead Articles Hogging Channels

Some articles receive a channel but never generate traffic. These articles hold their channel indefinitely, preventing it from being used by an article that could actually earn money.

🔴 Problem 3: No Clean Attribution

Without systematic tracking of which channel was assigned to which article and when, revenue reporting is unreliable. We can't confidently say "Article X earned $Y" because channel assignments are inconsistent and untracked.

🔴 Problem 4: Manual Process

Previously, channel assignments were done manually or semi-manually. At 10,000 articles/month, this doesn't scale. Human error leads to missed assignments, duplicate assignments, and slow response to channel status changes.

💡 The Solution

This system automates the entire channel lifecycle — from assignment to revenue tracking to reclamation. No human intervention needed for normal operations.

Three Parallel Systems

1
Channel Watcher Continuously monitors all 2,000 channels. The instant a channel changes state — goes idle, gets disapproved, comes back active — the system reacts. Idle channels enter the assignment queue. Disapproved channels trigger alerts. No polling delay; event-driven via BullMQ.
2
Auto-Assignment Engine When a new article is published, the Matching Engine instantly finds the best idle channel. "Best" means the channel that has been idle the longest (fair round-robin). The assignment is recorded with a precise timestamp. If no channel is available, the article enters a waiting queue and gets assigned the instant one frees up.
3
Revenue Attribution + Expiry Every 15 minutes, the system pulls revenue data from the AFS Reporting API and matches it to assignments using timestamps. Every hour, it checks for articles that have been live for 3 days with zero traffic — these get their channels reclaimed and the freed channel is immediately available for a new article.

🧠 How the System Thinks

The core insight is timestamp-based attribution. Every assignment is a time window:

The Attribution Rule

Article X was assigned Channel C at 2026-04-01 10:00:00.
Article X was unassigned from Channel C at 2026-04-04 10:00:00.

Any revenue earned by Channel C between those two timestamps belongs to Article X.

This is not a guess, not an estimate, not a statistical model. It's a deterministic record.

The system enforces a strict rule: one channel, one article, at any given time. A channel cannot be assigned to two articles simultaneously. This guarantees clean attribution — there's never ambiguity about which article earned the revenue.

Decision Tree

New article arrives │ ├── Are there idle channels? ── YES ──→ Assign longest-idle channel │ NO ──→ Add to waiting queue │ Channel becomes idle │ ├── Are articles waiting? ──── YES ──→ Assign to oldest waiting article │ NO ──→ Channel waits in idle queue │ Revenue data arrives (every 15 min) │ ├── Channel has assignment? ── YES ──→ Attribute revenue to that article │ NO ──→ Flag as orphan, alert team │ Article hits day 3 (hourly check) │ ├── Has any impressions? ───── YES ──→ Keep channel assigned │ NO ──→ Reclaim channel, expire article

⚙️ Tech Stack & Decisions

ComponentChoiceWhy ThisWhy Not Alternatives
Database PostgreSQL 16 ACID transactions, materialized views for analytics, rich indexing, battle-tested at this scale MySQL lacks materialized views. MongoDB is wrong fit for relational assignment data.
Cache / Queue Backend Redis 7 Sorted sets for O(log N) queue ops, sub-ms lookups, BullMQ backend, single dependency for both state and jobs Memcached has no sorted sets. Dedicated message queue adds ops overhead.
Job Queue BullMQ Redis-native, repeatable jobs, retries with backoff, rate limiting, concurrency control, dashboard available Kafka is massive overkill at 2K channels. RabbitMQ adds another service. BullMQ uses existing Redis.
Runtime Node.js Non-blocking I/O perfect for queue workers + API. Large ecosystem. Same language front-to-back. Python would work but Node's event loop is naturally suited for queue-worker patterns.
API Express Minimal, fast, well-understood. No magic. Fastify is slightly faster but Express ecosystem is larger. Doesn't matter at our request volume.
Dashboard Vanilla HTML/CSS/JS Zero build step, zero dependencies, served by Express directly. Fast to iterate. React/Vue would add build complexity for a relatively simple dashboard.
What We Deliberately Excluded

Kafka — We process ~700 events/day (10K articles + channel changes + revenue pulls). BullMQ handles this trivially. Kafka's operational overhead isn't justified.

ClickHouse — Our analytical queries run against ~10K-50K rows. PostgreSQL with materialized views returns in <100ms. ClickHouse shines at billions of rows.

Kubernetes — Single Node.js process handles everything. PM2 for process management is sufficient.

We can add any of these when measured performance requires it. See Scaling Path.

🔄 Flow 1: Auto-Assignment

This is the core flow. When a new article is published, it needs a channel immediately so it can start earning revenue from the first impression.

Step-by-Step Walkthrough

1
Article Creation (API Layer) Your publishing system sends a POST to /api/articles with the article's ID, URL, category, and publish date. The API inserts the article into PostgreSQL with status = "pending" and immediately adds a job to the article-assignment BullMQ queue. The API returns 201 within milliseconds — it doesn't wait for assignment.
2
Job Pickup (Matching Engine Worker) The Matching Engine worker is always listening on the article-assignment queue. It picks up the job typically within 10-50ms. It extracts the articleId from the job data.
3
Article Validation The worker queries PostgreSQL to verify the article exists and is in a valid state (pending or assigned). If the article doesn't exist or is in an invalid state (e.g., already expired), the job is skipped with a logged reason.
4
Channel Selection (Redis ZPOPMIN) The worker calls ZPOPMIN on the Redis sorted set ca:idle_channels. This atomically removes and returns the channel with the lowest score (= the oldest idle_since timestamp). This ensures longest-idle-first assignment — fair rotation so no channel is favored or neglected.

Why ZPOPMIN? It's atomic — in a concurrent environment, two workers can't grab the same channel. It's O(log N) — fast even with 2,000 channels.
5
If a channel is available:
  • PostgreSQL: Create assignment record (article_id, channel_id, assigned_at = NOW(), status = "active")
  • PostgreSQL: Update channel status to "assigned", set assigned_to = article_id
  • PostgreSQL: Update article status to "assigned"
  • Redis: Set ca:channel_assignment:{channelId} → articleId
  • Redis: Set ca:article_channel:{articleId} → channelId
  • PostgreSQL: Write to channel_log (event = "assigned", article_id, timestamp)
All PostgreSQL writes happen in a single operation sequence. If any step fails, the channel is returned to the idle queue.
6
If NO channel is available (idle queue empty): The article is added to the Redis sorted set ca:waiting_articles, scored by the current timestamp (FIFO ordering). When any channel becomes idle in the future, the system immediately checks this waiting queue and assigns automatically. The article's status remains "pending".

Edge Cases & Error Handling

ScenarioWhat HappensRecovery
Article doesn't exist in DB Job skipped, logged as "article_not_found" No retry needed — likely a race condition or invalid input
Article already has active assignment Job skipped, logged as "already_assigned" Duplicate job protection
Redis ZPOPMIN returns empty (no idle channels) Article added to waiting queue Assigned automatically when any channel becomes idle
PostgreSQL write fails after channel popped from Redis Channel is pushed back to idle queue (ZADD) BullMQ retries the job (3 attempts, exponential backoff)
Redis state update fails after PostgreSQL write succeeds PostgreSQL is source of truth — next sync-redis.js run fixes Redis Run node scripts/sync-redis.js or wait for next restart
Worker crashes mid-job BullMQ marks job as stalled, re-queues after visibility timeout Automatic — BullMQ handles this
Two articles arrive simultaneously BullMQ processes with concurrency=1, so they're serialized No race condition — ZPOPMIN is atomic, concurrency=1 is sequential
Why Concurrency = 1?
The Matching Engine runs with concurrency: 1 to prevent race conditions. If two jobs ran simultaneously, they could both try to ZPOPMIN the same channel (unlikely but possible with Redis pub/sub delays) or create conflicting assignments. Sequential processing at our volume (~330 articles/day = ~14/hour) adds negligible latency.

📡 Flow 2: Channel State Changes

Channels change state for various reasons — Google disapproves them, they stop delivering, they get reactivated. The system must react to each transition correctly.

Channel Status Lifecycle

┌─────────────────────────┐ │ │ ┌──────────────▼──────────────┐ │ │ IDLE │ │ │ In Redis idle queue │ │ │ Available for assignment │ │ └──────────────┬──────────────┘ │ │ │ │ Matching Engine assigns │ ▼ │ ┌──────────────────────────────┐ │ │ ASSIGNED │ │ │ Earning revenue for article │ │ │ Active assignment exists │ │ └───┬──────────┬──────────┬───┘ │ │ │ │ │ │ │ │ Article │ │ │ │ expires │ │ │ │ (3-day) │ │ │ ▼ │ │ │ Return to IDLE ───────┘ │ │ │ │ Manual free / │ │ channel goes idle │ ▼ │ Return to IDLE ──────────────────┘ │ │ Google disapproves ▼ ┌──────────────────────────────┐ │ DISAPPROVED │ │ NOT in idle queue │ │ Cannot be assigned │ │ Slack alert sent │ │ Requires manual review │ └──────────────┬───────────────┘ │ │ Manual reactivation ▼ Return to IDLE ──────────────────┘

When a Channel Goes Idle

1
Trigger: PUT /api/channels/:id/status with {"status": "idle"}, or internal event from channel monitoring system.
2
Close Active Assignment: If the channel has an active assignment, it's closed — unassigned_at = NOW(), status = "completed". The attribution window for the previous article is now sealed. Any revenue earned after this point will be attributed to the next article.
3
Update State: Channel status → "idle" in PostgreSQL. idle_since = NOW(). Added to Redis idle sorted set with current timestamp as score.
4
Check Waiting Queue: Immediately check ca:waiting_articles — if any articles are waiting for a channel, pop the oldest one and trigger assignment. This means a freed channel can be reassigned within milliseconds.

When a Channel Gets Disapproved

1
Close Active Assignment: Same as idle transition — any active assignment is closed cleanly.
2
Skip Idle Queue: Disapproved channels are NOT added to the idle queue. They can't serve ads, so assigning them would waste an article's time.
3
Alert: Slack notification sent with channel ID and details. Someone needs to review and either fix the issue or permanently retire the channel.
4
Article Reassignment: The article that lost its channel gets a new assignment job queued. If idle channels are available, it gets reassigned immediately.

💰 Flow 3: Revenue Attribution

This is how we know exactly what each article earned. Every 15 minutes, the system pulls revenue data from the AFS Reporting API and matches it to article assignments.

AFS Pull Cycle

1
BullMQ Trigger: A repeatable job fires every 15 minutes on the revenue-attribution queue. This is configured at startup and persists across restarts (idempotent upsert).
2
AFS API Call: The worker calls the AFS Reporting API requesting revenue data by channel for the last 30 minutes. Why 30 minutes when we pull every 15? Overlapping windows ensure we never miss data — if a pull is delayed or fails, the next pull's wider window catches it.
3
Response Processing: The AFS API returns data in the format: [{ channelId, impressions, clicks, revenue, periodStart, periodEnd }]. Each entry represents one channel's performance for one time period.

Revenue Matching

4
Channel → Article Lookup: For each channel with revenue, the worker checks Redis (ca:channel_assignment:{channelId}) to find which article currently owns this channel. If Redis misses, it falls back to PostgreSQL (query active assignment for this channel).
5
Revenue Record Upsert: For each channel-period combination, the worker does an INSERT ... ON CONFLICT DO UPDATE on revenue_events. The unique key is (channel_id, period_start). This means:
  • First time seeing this data: INSERT — new revenue record created
  • Overlapping pull sees same data again: UPDATE — existing record refreshed (no duplication)
  • AFS corrects historical data: UPDATE — record is updated with latest figures
6
Materialized View Refresh: After all revenue records are written, both materialized views are refreshed CONCURRENTLY (non-blocking). The dashboard immediately reflects updated per-article and per-channel revenue.

Revenue Edge Cases

Edge CaseWhat HappensWhy
Channel reassigned mid-period Revenue for the period before unassigned_at → old article. Revenue after new assigned_at → new article. Assignment timestamps define attribution windows precisely
Gap between unassign and reassign Revenue during the gap is written with attributed = FALSE No article to attribute to — flagged for investigation
Channel earns revenue but has no assignment Written as orphan revenue (article_id = NULL). Slack alert sent. Shouldn't happen normally — indicates a system issue or manual channel use outside the system
AFS pull fails BullMQ retries 3 times with exponential backoff. If all fail, Slack alert. Overlapping windows mean next successful pull catches the missed data
AFS returns corrected data for past period UPSERT updates the existing record ON CONFLICT DO UPDATE on (channel_id, period_start) handles this automatically
AFS reports revenue for disapproved channel Written with attributed = FALSE if no active assignment exists Channels can earn residual revenue even after disapproval
Multiple articles in one 15-min period Each assignment has precise timestamps — revenue is matched to whichever was active during that period Timestamp-based, not period-based attribution
Deduplication Guarantee
The UNIQUE(channel_id, period_start) constraint plus ON CONFLICT DO UPDATE means you can pull the same AFS data 100 times and the revenue_events table will have exactly one record per channel-period. No duplicate counting. Ever.

⏰ Flow 4: 3-Day Expiry

Not every article earns money. Some get zero traffic. These articles are wasting a channel that could be earning money for a different article. The Expiry Worker identifies and reclaims these.

Detection Logic

Every hour, the Expiry Worker runs a SQL query:

SELECT a.*
FROM articles a
LEFT JOIN revenue_events r ON r.article_id = a.id
WHERE a.status IN ('assigned', 'active')
  AND a.published_at <= NOW() - INTERVAL '72 hours'
  AND a.published_at >= NOW() - INTERVAL '96 hours'
GROUP BY a.id
HAVING COALESCE(SUM(r.impressions), 0) = 0
   AND COALESCE(SUM(r.clicks), 0) = 0

Why 72-96 hours specifically?

  • 72 hours minimum: Give every article a fair 3-day window. Some articles have slow starts — traffic campaigns take time to ramp up.
  • 96 hours maximum: Don't keep re-checking old articles. Once an article passes the 96-hour mark, it's no longer checked by this query. If it had zero traffic at 72h and still zero at 96h, it was checked during that window and expired.
  • Hourly checks within the 72-96h window: The worker runs every hour, so an article published at midnight on Monday will be checked starting Thursday midnight through Friday midnight — up to 24 checks within the window.
Important: Zero Means Zero
"Zero traffic" means literally zero impressions AND zero clicks across the entire 3-day period. Even 1 impression means the article stays. This is intentionally aggressive — a channel earning even a fraction of a cent is better than a channel tied to a truly dead article.

Reclamation Process

For each zero-traffic article found:

1
Mark Article Expired: status = "expired", expired_at = NOW(), expiry_reason = "zero_traffic". The article is done — it can't receive a new channel.
2
Close Assignment: unassigned_at = NOW(), status = "expired". The attribution window is sealed. Historical data is preserved — we know this channel was assigned to this article from X to Y and earned $0.
3
Free Channel: Channel status = "idle", idle_since = NOW(). Pushed back into Redis idle sorted set.
4
Immediate Reassignment Check: The worker checks ca:waiting_articles. If articles are waiting, the freed channel is assigned immediately. A channel reclaimed from a dead article can be earning money for a new article within seconds.

⏳ Flow 6: Waiting Queue

When all channels are in use and a new article arrives, it enters the waiting queue. This ensures no article is lost — it will be assigned the moment a channel frees up.

How It Works

  • Data Structure: Redis sorted set ca:waiting_articles, scored by timestamp
  • Ordering: FIFO — oldest waiting article gets the first available channel
  • Trigger: Every time a channel goes idle (via Channel State Worker or Expiry Worker), the waiting queue is checked
  • Size: If the queue grows beyond 50 articles, a Slack alert fires — indicates channel shortage

Scenarios

ScenarioOutcome
Queue is empty, channel goes idleChannel stays in idle queue until an article arrives
Queue has 5 articles, 1 channel goes idleOldest article gets the channel. 4 remain waiting.
Queue has 10 articles, 3 channels go idle simultaneously3 oldest articles each get a channel. 7 remain waiting.
Article has been waiting 2+ hoursStill waiting — no timeout. Gets assigned when a channel frees up.

🗄️ Database: assignments

CREATE TABLE assignments (
    id              BIGSERIAL PRIMARY KEY,
    article_id      BIGINT NOT NULL REFERENCES articles(id),
    channel_id      BIGINT NOT NULL REFERENCES channels(id),
    assigned_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    unassigned_at   TIMESTAMPTZ,    -- NULL while active
    status          VARCHAR(20) NOT NULL DEFAULT 'active',
                    -- active | completed | expired
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
This is the attribution table
Every row defines a time window: channel C was assigned to article A from assigned_at to unassigned_at. All revenue for C during that window belongs to A. When unassigned_at is NULL, the assignment is still active.

🗄️ Database: revenue_events

CREATE TABLE revenue_events (
    id              BIGSERIAL PRIMARY KEY,
    channel_id      BIGINT NOT NULL REFERENCES channels(id),
    article_id      BIGINT,                    -- NULL if unattributed
    assignment_id   BIGINT REFERENCES assignments(id),
    impressions     INTEGER NOT NULL DEFAULT 0,
    clicks          INTEGER NOT NULL DEFAULT 0,
    revenue         NUMERIC(10, 4) NOT NULL DEFAULT 0,
    period_start    TIMESTAMPTZ NOT NULL,
    period_end      TIMESTAMPTZ NOT NULL,
    pulled_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    attributed      BOOLEAN NOT NULL DEFAULT TRUE,
    UNIQUE(channel_id, period_start)           -- Dedup key
);

The UNIQUE(channel_id, period_start) constraint is critical — it prevents duplicate revenue records when overlapping AFS pulls fetch the same data.

🗄️ Materialized Views

mv_revenue_per_article

Pre-aggregated revenue per article. Refreshed every 15 minutes. Powers the dashboard revenue table and /api/revenue/by-article.

Contains: article_id, external_id, url, category, published_at, status, total_impressions, total_clicks, total_revenue, rpm

mv_revenue_per_channel

Pre-aggregated revenue per channel. Shows total articles served, total revenue, etc.

Why Materialized Views?
Aggregating revenue across thousands of events for every dashboard load would be expensive. Materialized views pre-compute the aggregations and are refreshed every 15 minutes (when new revenue data arrives). Dashboard queries hit the pre-computed view — O(1) per article instead of scanning revenue_events.

🔴 Redis: Idle Channel Queue

KeyTypeOperations
ca:idle_channels Sorted Set ZADD — add channel (score = idle_since timestamp in ms)
ZPOPMIN — get & remove longest-idle channel (atomic)
ZREM — remove specific channel
ZCARD — count idle channels
ZRANGE — list idle channels with scores

Why a sorted set? We need to always assign the longest-idle channel first (fair rotation). Sorted sets maintain order by score. ZPOPMIN atomically removes and returns the lowest-scored member — perfect for "give me the channel that's been idle the longest".

🔴 Redis: Waiting Article Queue

ca:waiting_articles Sorted Set Same operations as idle queue but for articles waiting for channels. FIFO ordering by timestamp.

🔴 Redis: Assignment State Store

Key PatternTypePurpose
ca:channel_assignment:{channelId} String Maps channel → article. O(1) lookup during revenue attribution. "What article does channel 1051 belong to?"
ca:article_channel:{articleId} String Reverse mapping: article → channel. "What channel is article 9050 using?"

Both mappings are set when an assignment is created and deleted when it's closed. They always match PostgreSQL — Redis is the fast lookup layer.

🔴 Redis: Recovery from Loss

If Redis crashes and loses all data:

node scripts/sync-redis.js

This script reads all idle channels and active assignments from PostgreSQL and rebuilds the Redis state. Takes seconds for 2,000 channels. The system is fully operational again immediately after.

PostgreSQL is always the source of truth
Redis is a performance optimization, not a primary data store. Losing Redis means temporary inability to process jobs (BullMQ needs Redis). But no data is lost — PostgreSQL has everything. Rebuild Redis from PG and you're back in business.

🔴 Redis: BullMQ Queues

BullMQ uses Redis for queue state, job data, and scheduling. Four queues are defined:

QueueTypeConfig
article-assignmentEvent-drivenConcurrency: 1, Retries: 3, Backoff: exponential
channel-stateEvent-drivenConcurrency: 1, Retries: 3, Backoff: exponential
revenue-attributionRepeatable (15 min)Concurrency: 1, Retries: 3
article-expiryRepeatable (1 hour)Concurrency: 1, Retries: 3

Repeatable jobs are configured using BullMQ's upsertJobScheduler — idempotent, so restarting the app doesn't create duplicate schedules.

⚙️ Worker: Matching Engine

File: src/workers/matchingEngine.js

Queue: article-assignment

Trigger: New article created via API

Job Data Format

{
  "articleId": 9050,
  "externalId": "my-article-slug"  // For logging
}

Processing Logic

  1. Validate article exists and is in valid state
  2. ZPOPMIN from ca:idle_channels
  3. If channel found: create assignment in PG, update Redis state, log event
  4. If no channel: add to ca:waiting_articles
  5. On any failure: return channel to idle queue (ZADD)

Concurrency: 1

Critical for correctness. If two jobs ran simultaneously, they could create conflicting assignments. At ~14 articles/hour, sequential processing adds no meaningful latency.

⚙️ Worker: Channel State

File: src/workers/channelState.js

Queue: channel-state

Trigger: Channel status change via API

Handles Three Transitions

  • → idle: Close assignment, add to idle queue, check waiting articles
  • → active: Remove from idle queue, update PG
  • → disapproved: Close assignment, skip idle queue, Slack alert

⚙️ Worker: Revenue Attribution

File: src/workers/revenueAttribution.js

Queue: revenue-attribution

Trigger: Every 15 minutes (repeatable job)

Process

  1. Call AFS Reporting API (30-min window for overlap safety)
  2. For each channel with revenue: look up assignment
  3. UPSERT revenue_events (dedup on channel_id + period_start)
  4. Flag unattributed revenue
  5. Refresh materialized views
  6. Alert on orphan revenue

⚙️ Worker: Expiry

File: src/workers/expiryWorker.js

Queue: article-expiry

Trigger: Every 1 hour (repeatable job)

Process

  1. Query PG: articles published 72-96h ago with zero impressions/clicks
  2. For each: mark expired, close assignment, free channel
  3. Check waiting queue for immediate reassignment

🔧 Failure & Recovery

FailureImpactRecovery
Redis crashes No new assignments. No job processing. Existing assignments and revenue records in PG are safe. Restart Redis. Run node scripts/sync-redis.js. System resumes.
PostgreSQL crashes No writes possible. API returns errors. Workers fail. Restart PG. Workers automatically retry failed jobs (BullMQ backoff).
Node.js process crashes No API. No workers. Pending BullMQ jobs accumulate in Redis. PM2 auto-restarts. Jobs in Redis are processed on restart.
AFS API unavailable Revenue data not pulled. No attribution for missed periods. BullMQ retries 3x. Next successful pull's 30-min window catches up.
Redis + PG state diverge Redis shows different idle channels than PG. Run node scripts/sync-redis.js — rebuilds Redis from PG truth.
